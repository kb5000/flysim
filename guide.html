CHAPTER 0

教学大纲

本章目标

  • 知道最终会做出什么，而不是一开始就被 3D 数学和空气动力学吓住。
  • 准备一个可以运行 ES Modules 的本地开发环境。
  • 理解本课程采用“每章都能运行”的增量开发方式。

开始前需要会什么

你只需要理解 JavaScript 的变量、函数、数组、对象、类和 import/export。不要求学过 WebGL、矩阵、微积分或空气动力学； 遇到这些概念时，本教程会先解释直觉，再给公式和代码。

先把项目运行起来

  1. 启动静态服务器

    在项目根目录执行：

    python -m http.server 8000
    浏览器出于安全原因，不允许从 file:// 页面自由加载 ES Module。HTTP 服务器还能让 Gamepad、音频和模块错误更接近真实部署环境。

  2. 打开模拟器与教材

    http://localhost:8000/
http://localhost:8000/guide.html
    模拟器应显示天空、跑道、地形和 HUD；教材页应显示左侧章节目录。

  3. 打开开发者工具

    在 Chrome 或 Edge 中按 F12，固定保留 Console 与 Network 面板。 从这一刻开始，每完成一步都先看控制台是否有红色错误。

学习方法

不要一次复制完整项目。每章都建议新建一个临时分支，先实现最小版本， 运行并观察，再与当前源码比较。飞行模拟器是多个小系统的组合； 如果跳过中间检查，最后看到的通常只是一块黑屏或一架无法控制的飞机。

本教程不按文件名字母顺序阅读，而按依赖关系展开。先理解每一帧如何推进， 再学习数学与渲染基础，最后进入飞行动力学和设备输入。这样每个公式都能落到代码用途上。

第一阶段：骨架HTML 入口、模块装配、固定步长与渲染插值。

CHAPTER 1

系统架构：数据如何穿过项目

本章目标

  • 画出 FlySim 的数据流图。
  • 理解“状态”和“控制命令”为什么必须分开。
  • 亲手搭出一个不含任何飞行公式的项目骨架。

第一步：只创建页面骨架

初学者最常见的错误，是一开始就写物理。正确顺序是先建立可以装配模块的容器。 在最小版本的 index.html 中放两个 Canvas： 

<div id="app">
  <canvas id="gl"></canvas>
  <canvas id="hud"></canvas>
</div>
<script type="module" src="src/main.js"></script>

gl 负责 3D，hud 负责文字和仪表。分成两个画布不是必须， 但会极大简化字体、线条、裁剪和界面布局。

WebGL 很擅长大量三角形，却不擅长直接画清晰文字；Canvas 2D 很擅长仪表，却不适合成千上万的 3D 顶点。让工具各做擅长的事。

第二步：定义两类数据

const state = {
  pos: [0, 0, 0],
  vel: [0, 0, 0],
  q: [0, 0, 0, 1],
  omega: [0, 0, 0],
};

const ctrl = {
  aileron: 0,
  elevator: 0,
  rudder: 0,
  throttle: 0,
};

state 回答“飞机现在是什么状态”，它由物理系统写入； ctrl 回答“驾驶员希望飞机做什么”，它由输入系统写入。 渲染系统只能读取二者，不应偷偷修改它们。

常见错误：在键盘事件里直接移动飞机

例如按 W 就执行 state.pos[1] += 1。这样速度依赖键盘重复率， 物理无法计算惯性，手柄也要重写一套逻辑。输入应该只改变 ctrl， 位置只能由模拟器积分得到。

第三步：用接口而不是具体设备连接系统

键盘和手柄都输出同样的对象，固定翼和四轴都读取同样的对象。 因此添加设备或机型时，变化局限在一侧。 

const input = selectedDevice.sample(dt);
applyControls(ctrl, input);

if (state.aircraftType === 'quad') {
  stepQuad(state, ctrl, fixedDt);
} else {
  stepPhysics(state, ctrl, fixedDt);
}

动手练习

新增一个虚拟“自动驾驶输入”，让它始终返回 {aileron:0,elevator:0,rudder:0,throttleDelta:0}。 如果无需修改物理模块就能选择它，说明分层正确。

FlySim 的核心设计是“统一控制量 + 可切换模拟器 + 共享渲染状态”。 键盘和手柄最终都产生同一种 ctrl，固定翼和四轴各自解释它， 然后把位置、姿态和遥测写回 state。

Keyboard / Gamepad→统一 ctrl→ Fixed-wing / Quad physics→state→ Scene + Camera + HUD + Audio

分层原则

目录职责不应负责
src/sim状态、力、力矩、积分、接地DOM、WebGL、按键编号
src/input设备读取并输出统一控制量直接修改飞机位置
src/engine通用数学与 WebGL 基础设施知道 C172 参数
src/world地形、跑道、天空决定油门或舵面
src/render场景和程序机模计算真实气动力
src/uiHUD、地图、音频反馈成为物理状态真源

为什么统一控制量很重要

aileron、elevator、rudder 和 throttle 是项目内部协议。固定翼把前三者解释为舵面指令； 四轴把它们解释为滚转、俯仰、偏航角速度目标。输入层无需知道当前飞行器类型， 因而切换机型不会复制设备代码。

CHAPTER 2

主循环：为什么物理不能直接使用帧间隔

本章目标

  • 先写出一个最小动画循环。
  • 亲眼看到可变步长带来的不稳定，再改成固定步长。
  • 实现位置线性插值与姿态球面插值。

从最小动画循环开始

let last = performance.now() / 1000;

function frame(nowMs) {
  const now = nowMs / 1000;
  const dt = now - last;
  last = now;

  update(dt);
  render();
  requestAnimationFrame(frame);
}

requestAnimationFrame(frame);

这段代码适合动画，却不适合刚体物理。浏览器卡顿时 dt 可能从 0.016 秒跳到 0.2 秒。欧拉积分会一步跨过地面，弹簧力也可能大到让飞机爆飞。

做一个能看见问题的小实验

velocity += gravity * dt;
height += velocity * dt;

在开发者工具中开启 CPU throttling。你会发现同一个物体落地时间和穿透程度改变。 这说明“显示帧率”不应该决定“物理时间离散方式”。

一步步加入累加器

  1. 选择固定步长

    const FIXED_DT = 1 / 120;。120 Hz 对快速角运动和起落架弹簧更稳定。

  2. 保存尚未模拟的时间

    每帧执行 acc += dt，不要立即把全部时间塞进一次物理更新。

  3. 按固定小步消耗时间

    while (acc >= FIXED_DT) {
  previous = snapshot(state);
  step(state, ctrl, FIXED_DT);
  acc -= FIXED_DT;
}

  4. 插值显示状态

    物理可能落后渲染不足一个步长。用 alpha=acc/FIXED_DT 在前后状态间插值。

物理使用固定步长是为了稳定和可重复；渲染插值是为了不让固定步长在高刷新率屏幕上显得抖动。这两个目标不能混为一谈。

防止“死亡螺旋”

如果一帧很慢，物理要补很多步；补步又让下一帧更慢。项目把帧间隔限制为 0.25 秒， 每帧最多执行 8 步。代价是严重卡顿时模拟时间会稍慢，但页面不会永久冻结。

切换到 60 Hz、120 Hz 显示器或短暂拖动窗口时，飞机轨迹应基本相同，画面仍平滑。

浏览器的 requestAnimationFrame 可能以 60、120 或不稳定帧率运行。 若直接用每帧 dt 积分，物理结果会随显示器和卡顿变化。 因此 src/main.js 使用累加器执行固定的 1/120 s 物理步。

固定步长累加器 A ← A + Δtframe 当 A ≥ h 时：执行 physics(h)，然后 A ← A − h 渲染插值系数：α = A / h

这里的 h 是固定物理步长。每次物理更新前保存旧姿态， 渲染时用 lerp 插值位置、用 slerp 插值四元数。 这样物理保持确定性，画面仍然平滑。 

while (acc >= FIXED_DT) {
  prevState = snapshot(state);
  stepPhysics(state, ctrl, FIXED_DT);
  acc -= FIXED_DT;
}
render(interpolate(prevState, state, acc / FIXED_DT));

每帧的完整顺序

  1. 计算并限制帧间隔，避免切回标签页后一次推进过远。
  2. 读取当前选中的输入设备，累积油门与配平。
  3. 消费按键边沿动作，例如襟翼、暂停、视角和重置。
  4. 按当前机型调用固定翼或四轴物理。
  5. 插值位置和姿态，推进螺旋桨视觉角度。
  6. 更新相机，绘制 3D 场景、HUD，再更新声音参数。

CHAPTER 3

坐标系与数学基础

本章目标

  • 不用死记公式，也能判断一个向量属于世界系还是机体系。
  • 实现最小 vec3 工具并理解点积、叉积。
  • 理解四元数在项目中解决的具体问题。

先用纸画坐标轴

在纸上画一架机头向北的飞机。世界系的“北”永远是 +Y；飞机转弯后， 机体系的“前方”仍是 +X，但它在世界系中可能指向任意方向。 因此一句“推力沿 +X”是不完整的，必须说明是“机体系 +X”。

调试规则

每个变量名或注释都应能回答它属于哪个坐标系。推荐后缀： Fbody、Fworld、vBody、 omegaW。方向错误时，先查坐标系，再查正负号。

亲手写三个向量函数

function add(a, b) {
  return [a[0] + b[0], a[1] + b[1], a[2] + b[2]];
}

function dot(a, b) {
  return a[0]*b[0] + a[1]*b[1] + a[2]*b[2];
}

function cross(a, b) {
  return [
    a[1]*b[2] - a[2]*b[1],
    a[2]*b[0] - a[0]*b[2],
    a[0]*b[1] - a[1]*b[0],
  ];
}

点积回答“两个方向有多同向”，所以可把轮胎速度投影到滚动方向。 叉积回答“绕哪个轴旋转、强度多大”，所以力臂和力的叉积就是力矩。

从二维旋转理解四元数

二维只需一个角度；三维需要旋转轴和角度，而且旋转顺序会影响结果。 欧拉角连续积分还会在俯仰接近 90° 时丢失一个自由度。 四元数保存的是一个整体旋转，不依赖“先滚转还是先偏航”的人为拆分。

  1. 先实现轴角四元数

    单位轴 a 和角度 θ 对应 (a sin(θ/2), cos(θ/2))。

  2. 实现四元数乘法

    乘法用于组合旋转。顺序不可交换：a⊗b 通常不等于 b⊗a。

  3. 实现向量旋转

    理论形式是 q(v,0)q*；项目使用展开后的快速公式。

  4. 每步归一化

    浮点误差会破坏单位长度，归一化是姿态积分不可省略的维护操作。

动手练习

用 quat.fromHeadingPitchRoll(π/2,0,0) 创建朝东姿态， 再旋转机体前向量 [1,0,0]。结果在 ENU 世界系中应接近 [1,0,0]。

两个坐标系

世界坐标 ENU

+X 向东，+Y 向北，+Z 向上。位置、世界速度、地形都使用它。

机体坐标

+X 机头，+Y 右翼，+Z 向下。这是航空动力学常用约定。

姿态四元数 s.q 表示“机体系向量旋转到世界系”。因此机体推力先在机体系构造， 再用 quat.rotate(s.q, forceBody) 变换到世界系；世界速度转换为空速时使用逆旋转。

向量运算

点积：a · b = axbx + ayby + azbz 叉积：τ = r × F 长度：‖v‖ = √(vx² + vy² + vz²)

点积用于投影轮胎速度，叉积用于“力臂 × 力 = 力矩”，归一化用于方向向量。 src/math.js 的 vec3 保持纯函数风格，降低状态被意外修改的风险。

为什么使用四元数

欧拉角适合展示，却不适合连续积分，因为会出现万向节锁。四元数 q = (x, y, z, w) 可稳定组合旋转。机体系角速度 ω = (p,q,r) 的积分关系是：

姿态微分 q̇ = 1/2 · q ⊗ (ωx, ωy, ωz, 0) qnext = normalize(q + q̇Δt)

每次积分后归一化非常重要，否则浮点误差会让四元数逐渐失去单位长度， 旋转矩阵也会产生缩放和剪切。

矩阵与渲染

WebGL 使用列主序 4×4 矩阵。顶点最终经历 clip = projection · view · model · position。 法线不能直接乘模型矩阵；存在非均匀缩放时应使用模型矩阵左上 3×3 的逆转置， 这就是 normalFromMat4 的用途。

CHAPTER 4

WebGL2 渲染引擎

本章目标

  • 从空白 Canvas 画出第一个彩色三角形。
  • 理解顶点缓冲、索引、VAO、着色器各自保存什么。
  • 建立可复用的 Mesh，而不是每个物体手写 WebGL 调用。

步骤 1：确认 WebGL2 可用

const canvas = document.getElementById('gl');
const gl = canvas.getContext('webgl2');
if (!gl) throw new Error('WebGL2 unavailable');

先调用 gl.clearColor 和 gl.clear 把背景改成明显的蓝色。 如果颜色没有改变，先不要继续写着色器，因为 Canvas 尺寸、脚本路径或上下文创建已经有问题。

步骤 2：写最小着色器

// vertex shader
#version 300 es
layout(location=0) in vec3 aPos;
void main() {
  gl_Position = vec4(aPos, 1.0);
}

// fragment shader
#version 300 es
precision highp float;
out vec4 frag;
void main() {
  frag = vec4(0.3, 0.9, 0.6, 1.0);
}

顶点着色器对每个顶点运行，必须写出裁剪空间位置； 片元着色器对三角形覆盖的像素运行，写出颜色。 此时还没有相机，直接使用 −1 到 +1 的裁剪空间坐标。

步骤 3：上传三个顶点

const positions = new Float32Array([
  -0.6, -0.5, 0,
   0.6, -0.5, 0,
   0.0,  0.6, 0,
]);

const buffer = gl.createBuffer();
gl.bindBuffer(gl.ARRAY_BUFFER, buffer);
gl.bufferData(gl.ARRAY_BUFFER, positions, gl.STATIC_DRAW);
gl.enableVertexAttribArray(0);
gl.vertexAttribPointer(0, 3, gl.FLOAT, false, 0, 0);
gl.drawArrays(gl.TRIANGLES, 0, 3);
画布中央出现绿色三角形。若没有，检查着色器编译日志和 Canvas 实际宽高。

步骤 4：为什么要封装 Mesh

一个真实场景有地形、跑道、机身、舵面和旋翼。如果每次绘制都重新绑定三组属性， 代码会迅速失控。VAO 记住“属性位置如何解释缓冲”，Mesh.draw() 只需绑定 VAO 并调用 drawElements。

步骤 5：加入模型、观察、投影矩阵

gl_Position = uViewProj * uModel * vec4(aPos, 1.0);

uModel 把零件局部坐标变到世界坐标； uViewProj 先把世界移到相机坐标，再做透视投影。 先只画一个单位盒，再移动相机；不要同时加入光照、雾和复杂机模。

动手练习

关闭 gl.CULL_FACE，再交换三角形两个顶点。观察画面为何不再受绕序影响，然后重新开启剔除。

初始化与面剔除

src/engine/gl.js 创建 WebGL2 上下文，开启深度测试与背面剔除。 项目约定逆时针顶点为正面，即 gl.frontFace(gl.CCW)。 地形或跑道三角形如果顺序写反，就会从上方被当成背面剔除。

着色器程序

src/engine/shader.js 完成“编译顶点着色器 → 编译片元着色器 → 链接程序 → 缓存 uniform/attribute 地址”。错误日志在编译阶段抛出， 让 GLSL 问题不会静默变成黑屏。

网格抽象

Mesh 把位置、法线、颜色分别上传到属性位置 0、1、2， 并把 VAO 保存为可重复绘制的状态集合。静态机模使用 STATIC_DRAW，不断重建高度的地形使用 DYNAMIC_DRAW 和 bufferSubData。

三角形法线方向 n = (b − a) × (c − a) 若 nz > 0，则从上方看 a→b→c 为逆时针。

光照与雾

场景片元着色器混合半球环境光和单方向漫反射。距离雾使用指数形式：

f = clamp(1 − e−ρd, 0, 1) C = mix(Clit, Cfog, f)

d 是相机到片元距离，ρ 是雾密度。指数雾在远距离逐渐趋近 1， 能自然隐藏有限地形窗口的边缘。

CHAPTER 5

程序世界：噪声、地形、跑道与天空

本章目标

  • 从一个平面网格开始，逐步变成程序地形。
  • 理解为什么物理碰撞和视觉地形必须调用同一个高度函数。
  • 实现不会无限占用显存的滑动地形窗口。

步骤 1：先生成完全平坦的网格

设网格有 nx × ny 个格子，则顶点数是 (nx+1)(ny+1)。每个格子拆成两个三角形： 

const a = j * stride + i;
const b = a + 1;
const c = a + stride;
const d = c + 1;
indices.push(a, b, c, b, d, c);

从 +Z 上方看，这个顺序是逆时针。先让所有高度为零，并给法线 [0,0,1]。确认整个平面可见后再加入噪声。

步骤 2：写可重复的随机函数

不能在每次采样时调用 Math.random()，否则同一地点每帧高度不同。 整数哈希把格点坐标映射到固定伪随机值，然后在四个格点间平滑插值。 

const ux = tX*tX*(3 - 2*tX);
const uy = tY*tY*(3 - 2*tY);
const lower = lerp(v00, v10, ux);
const upper = lerp(v01, v11, ux);
return lerp(lower, upper, uy);

步骤 3：从噪声得到山脉

单层噪声像缓慢起伏的布。叠加高频低振幅层后，低频决定山体， 高频决定岩石细节。先用两层观察，再增加到五层，避免不知道哪一层造成异常。

步骤 4：让地形跟随飞机

无限地形并不意味着无限顶点。将飞机坐标除以格子尺寸并取整， 得到当前中心格。只有中心格变化时才重写顶点缓冲。 

const centerI = Math.round(planeX / TILE);
const centerJ = Math.round(planeY / TILE);
if (centerI !== oldI || centerJ !== oldJ) {
  rebuild(centerI, centerJ);
}
窗口随飞机移动时，顶点使用真实世界坐标，而不是永远围绕原点。这样跑道、碰撞和小地图仍共享同一坐标。

步骤 5：共享 groundHeight

地形渲染、起落架碰撞、四轴接地和小地图都调用 groundHeight(x,y)。如果渲染自己保存一份高度数组而物理使用另一算法， 飞机会漂浮或陷入地面。

动手练习

把机场压平半径临时改为 0，观察跑道如何穿过山坡；再恢复半径并调整平滑过渡带，比较硬切与 smoothstep 的差异。

值噪声与 FBM

valueNoise 对网格四角生成确定性伪随机值，再使用平滑曲线 3t²−2t³ 做双线性插值。单层噪声过于柔和，因此 fbm 叠加多个频率层：

Fractal Brownian Motion H(x,y) = Σ aiN(fix, fiy) / Σai fi+1 = lacunarity · fi ai+1 = gain · ai

滑动高度场

地形不是无限网格，而是围绕飞机的 80×80 单元窗口，每格 50 m。 飞机跨过网格边界后，顶点位置、法线和颜色被重新采样并上传。 物理和渲染共享同一个 groundHeight(x,y)，因此视觉地面与碰撞地面一致。

高度场法线

相邻高度的中心差分近似坡度：

ñ = (hleft − hright, hdown − hup, 2Δ) n = ñ / ‖ñ‖

法线的 z 分量还用于估算坡度并压暗陡坡。机场附近通过平滑插值将高度压到零， 从而保证跑道平整又不产生突兀断崖。

天空为何只需一个三角形

src/world/sky.js 生成覆盖屏幕的超大三角形，不提交传统天空球顶点。 片元着色器用逆视图投影矩阵重建世界射线，根据射线的向上分量混合天顶、地平线和地雾， 再用 dot(viewDir, sunDir) 的高次幂绘制太阳盘与光晕。

固定翼六自由度动力学

本章目标

  • 从“自由落体盒子”逐步实现一架能起飞、配平、失速和着陆的固定翼。
  • 理解每个气动系数在代码中控制什么现象。
  • 学会用单位和数量级检查公式，而不是只靠试飞猜参数。

前置知识

完成本教程第 2、3 章，知道固定步长、机体系和世界系，并能使用向量加法、叉积与四元数旋转。 不需要预先学习流体力学。

里程碑 1：只实现重力

先把飞机当作没有空气的刚体：

const gravity = [0, 0, -mass * 9.80665];
const acceleration = scale(gravity, 1 / mass);
state.vel = addScaled(state.vel, acceleration, dt);
state.pos = addScaled(state.pos, state.vel, dt);
从空中生成时，飞机竖直加速下落。它不会向前飞，也不会改变姿态。
从最简单的力开始，可以先验证世界 +Z、重力符号、质量单位和积分器。如果这一步方向错误，加入升力后只会更难排查。

里程碑 2：加入机体前向推力

推力定义在机体系 +X。不能直接加到世界 X，因为飞机转向后机头不再朝东。 

const thrustBody = [thrustNewton, 0, 0];
const thrustWorld = quat.rotate(state.q, thrustBody);
forceWorld = add(forceWorld, thrustWorld);
水平生成的飞机会向机头方向加速，但仍然下落，因为还没有升力。

用牛顿第二定律估算：3000 N 推动 1100 kg 飞机，静止时仅推力贡献约 2.73 m/s²。若代码产生几十倍加速度，通常是把质量遗漏了。

里程碑 3：计算空气相对速度

物理状态中的速度在世界系，但气动公式需要机体系分量 (u,v,w)： 

const vBody = quat.rotateInv(state.q, state.vel);
const [u, v, w] = vBody;
const speed = Math.hypot(u, v, w);
const alpha = Math.atan2(w, u);
const beta = Math.asin(clamp(v / speed, -1, 1));

在本项目机体系中 +Z 向下。当飞机机头相对速度方向抬高时， 速度在机体系中出现正 w，所以正攻角来自 atan2(w,u)。

常见错误：把“相对风”与“飞机穿过空气的速度”混用

本代码把 vBody 当作飞机相对空气的速度方向，因此阻力沿其反方向。 如果你改成气流吹向飞机的“相对风”定义，攻角和力方向都要同时翻转。

里程碑 4：加入最小升力与阻力

  1. 计算空气密度和动压

    Q = 0.5 * rho * V * V，单位是 Pa，也就是 N/m²。

  2. 计算升力系数

    CL = CL0 + CLa * alpha。注意 CLa 的单位是“每弧度”，不是每度。

  3. 变成牛顿

    L = Q * S * CL。面积 S 把单位面积压力变成总力。

  4. 计算阻力

    先用常数 CD0，再加入诱导阻力 CL²/(πeAR)。

  5. 投回机体系

    升力垂直于速度，阻力反向于速度；项目在 X-Z 平面按攻角旋转组合。

const Fx = L * Math.sin(alpha) - D * Math.cos(alpha);
const Fz = -L * Math.cos(alpha) - D * Math.sin(alpha);
const forceBody = [Fx, sideForce, Fz];
给定足够前向速度后，飞机不再自由落体；改变攻角会改变升力，但姿态暂时仍不会自行变化。

里程碑 5：加入失速，而不是限制攻角

不要在超过 15° 时直接把攻角夹回 15°，那会让飞机永远保持最大升力。 正确做法是让升力曲线在峰值后下降，同时阻力显著增加。 

const sigma = clamp((abs(alpha) - stallAngle) / blendWidth, 0, 1);
const clAttached = clamp(CL0 + CLa * alpha, -CLmax, CLmax);
const clSeparated = Math.sin(2 * alpha);
const CL = lerp(clAttached, clSeparated, sigma);

这里不是高精度翼型模型，而是行为模型：小攻角保持线性，高攻角逐渐接近平板流动。 测试只要求 25° 的升力低于 13°，确保“拉得更多”不会永远“升得更多”。

里程碑 6：让飞机能够转动

到目前为止，飞机可能像一块始终保持方向的魔法滑板。现在加入滚转、俯仰、偏航力矩。 每个系数项都可理解为“某种状态对某个力矩的贡献”：

  • Cma < 0：正攻角产生低头力矩，提供纵向静稳定。
  • Cmq < 0：正俯仰角速度产生反向阻尼。
  • Clda：副翼产生滚转力矩。
  • Cndr：方向舵产生偏航力矩。
  • Cnda：副翼引起反向偏航。

先只启用舵面项，确认方向；再逐项加入静稳定和阻尼。一次打开全部项会让符号错误很难定位。

里程碑 7：加入配平和襟翼

配平在代码中作为“额外升降舵等效偏角”。它不是自动驾驶，而是改变松杆时的平衡点。 襟翼同时增加升力、阻力并改变俯仰力矩；只增加升力会让襟翼成为没有代价的作弊按钮。

里程碑 8：加入三点起落架

对每个轮子执行“位置 → 穿透 → 接触点速度 → 法向力 → 摩擦 → 力矩”。 先只实现垂直弹簧，让飞机站住；再加入滚动摩擦；最后加入刹车、侧向抓地和前轮转向。

飞机静止时三个轮子接触，满油门沿跑道加速；达到约 55 kt 后拉杆可离地；重着陆会触发坠毁。

如何调参而不迷路

  1. 先固定质量和几何，因为它们是其他系数的基准。
  2. 用起飞距离调静推力与地面阻力。
  3. 用巡航余功率调推力随空速衰减和寄生阻力。
  4. 用配平测试调 Cm0/Cma/Cmde。
  5. 用短时舵输入调操纵效能，再用阻尼项调回中后的衰减。
  6. 每改一组参数就运行测试，不同时改五组。

动手练习：做一个“无阻力世界”

临时把所有 CD 项设为零，观察飞机如何不断加速。 再只恢复 CD0 和诱导阻力，比较高速与高升力状态分别由哪一项限制。

状态与空速

状态由世界位置 p、世界速度 v、姿态 q 和机体角速度 ω=(p,q,r) 构成。 无风条件下，机体相对空气的速度就是世界速度旋转回机体系。

V = √(u² + v² + w²) α = atan2(w, u) β = asin(v / V) Q = 1/2 ρV²

攻角 α 描述机头弦线相对气流的上下夹角；侧滑角 β 描述气流从机体侧面进入的程度；动压 Q 是所有气动力随速度平方增长的来源。

升力与失速

CL,lin = CL0 + CLαα + ΔCL,flap L = QS CL

小攻角时使用线性升力曲线。超过失速攻角后，代码将线性模型平滑混合到 sin(2α) 平板模型，使升力峰值后下降，而不是无限增长。

阻力

CD = CD0 + CDgear + CDflap + CL²/(πeAR) + CDsep D = QS CD

第一组是寄生阻力，平方项是有限翼诱导阻力，失速分离项在高攻角快速增加。 e 是奥斯瓦尔德效率，AR=b²/S 是展弦比。

气动力矩

Cm = Cm0 + Cmαα + Cmqq̂ + Cmδeδe Cl = Clββ + Clpp̂ + Clrr̂ + Clδaδa + Clδrδr Cn = Cnββ + Cnrr̂ + Cnpp̂ + Cnδrδr + Cnδaδa

带帽角速度是无量纲量，例如 p̂=pb/(2V)。 这样同一组导数可以在不同速度下保持相似物理意义。最终力矩为 Mx=QSbCl、My=QScCm、Mz=QSbCn。

刚体积分

线运动使用半隐式欧拉：先更新速度，再用新速度更新位置。

a = F/m vn+1 = vn + aΔt pn+1 = pn + vn+1Δt

ṗ = [L − (Iz−Iy)qr] / Ix q̇ = [M − (Ix−Iz)rp] / Iy ṙ = [N − (Iy−Ix)pq] / Iz

CHAPTER 7

四轴动力学：复用输入，不复用飞行模型

本章目标

  • 从固定翼项目中分离出第二套模拟，而不复制主循环和输入。
  • 计算理论悬停油门。
  • 实现一个容易驾驶的角速度闭环，而不是直接把摇杆当力矩。

步骤 1：先只做垂直运动

创建质量、最大推力和油门响应三个参数。忽略姿态，只沿世界 +Z 施加向上推力： 

const thrust = maxThrust * throttleState;
const accelerationZ = thrust / mass - gravity;
velocityZ += accelerationZ * dt;
height += velocityZ * dt;

理论悬停点来自 T=mg。当前参数为 1.6 kg 和 32 N：

hover = 1.6 × 9.80665 / 32 ≈ 0.49
约 49% 油门悬停，低于它下降，高于它上升。若悬停点不是这个数量级，先查单位和重力符号。

步骤 2：让推力跟随姿态

四轴推力沿机体上方，也就是本项目机体系 −Z。将 [0,0,-T] 旋转到世界系。倾斜后总推力的一部分变成水平分量， 垂直分量减少，所以保持高度需要增加油门。

这一步自然产生“倾斜才能平移”的四轴行为，不需要额外写前进力。

步骤 3：先尝试直接力矩，再理解它为什么难开

torqueX = ctrl.aileron * maxTorque;

直接力矩模式意味着摇杆回中后力矩为零，但已有角速度不会立即停止。 初学者会发现四轴像在太空中旋转，需要反打摇杆才能刹住。

步骤 4：改为目标角速度

const targetRate = input * maxRate;
const error = targetRate - currentRate;
const torque = clamp(error * gain - damping * currentRate,
                     -maxTorque, maxTorque);

摇杆回中时目标角速度为零，控制器自动产生反向力矩制动。 这不是姿态自稳：如果飞机已经倾斜，它会停止继续翻滚，但保持当前倾角。

步骤 5：加入线性阻力和地面

没有空气阻力时，倾斜一会儿得到的水平速度会永久保持。 简化线性阻力 Fdrag=-cv 能让速度逐渐衰减。 地面先用中心高度约束完成最小版本，之后可扩展到四支脚接触。

步骤 6：接入机型分派

if (state.aircraftType === 'quad') {
  stepQuad(state, ctrl, FIXED_DT, groundHeight);
} else {
  stepPhysics(state, ctrl, FIXED_DT, groundHeight);
}

切换时必须重置位置、速度、姿态、油门状态和控制对象。 否则固定翼 80% 油门会直接带入四轴，切换瞬间就升空。

进阶练习：姿态自稳模式

将摇杆映射为目标滚转/俯仰角，先用姿态误差产生目标角速度， 再沿用现有角速度内环。这样形成“姿态外环 + 角速度内环”。

四轴模式仍接收固定翼命名的控制量，但语义变成： aileron→滚转角速度、elevator→俯仰角速度、 rudder→偏航角速度、throttle→总推力。

总推力

四个旋翼合并为沿机体向上方向（机体系 −Z）的总推力：

Fthrust,body = (0,0,−Tmaxx) Fworld = R(q)Fbody + (0,0,−mg) − cdv 悬停油门 ≈ mg/Tmax

角速度闭环

输入不是直接力矩，而是期望角速度。一个简化 P 控制器比较目标与当前角速度：

ωtarget = input · ωmax τ = clamp[K(ωtarget−ω) − cωω, −τmax, τmax]

这类似飞控的 rate mode：松杆时目标角速度回到零，因此旋转会停止， 但当前倾角不会自动回正。若要实现自稳模式，应增加姿态目标外环。

简化地面接触

四轴没有逐支脚弹簧，而是把机体中心限制在 ground + groundClearance 以上。接地时清除向下速度并快速衰减水平速度。 这是成本很低的近似，适用于小型演示；精细模拟可改为四个接触点。

Gamepad API 的现实问题

navigator.getGamepads() 返回稀疏数组，其中可能包含 null； 浏览器还可能要求用户先按一次手柄按钮。项目同时监听连接事件、轮询设备、 缓存已知设备，并提供手动刷新和权限诊断。

油门为什么是增量

键盘 W/S 和手柄 A/B 是按钮，没有绝对轴，因此输入层输出 throttleDelta，主循环把它积分到持久的 ctrl.throttle。 舵面则直接使用当帧值，松手立即回中。

CHAPTER 9

程序机模、相机、HUD 与声音

本章目标

  • 用基础盒体搭出可辨认的两种机模。
  • 正确实现“绕铰链旋转”的舵面动画。
  • 从一个十字准星逐步构建 HUD、小地图和程序声音。

机模步骤 1：只做静态轮廓

固定翼先用一个长盒表示机身、一个薄盒表示机翼、两个小盒表示尾翼。 不要一开始就做舵面。先从追逐相机确认 +X 真的是机头，+Y 真的是右翼。

机模步骤 2：将静态几何合并

每个盒子单独绘制会增加 draw call。将不动的零件合并为一个 Mesh； 只有升降舵、副翼、方向舵和螺旋桨保留独立 Mesh。

机模步骤 3：正确构造铰链

  1. 把舵面几何放到真实位置

    例如方向舵顶点本身就应位于机尾，而不是留在原点。

  2. 记录铰链点

    铰链点也使用机体局部坐标，例如尾部某个 [x,y,z]。

  3. 构造 T(p)RT(−p)

    该局部矩阵只旋转舵面，场景随后再乘飞机世界矩阵。

  4. 用小输入检查符号

    右方向舵时后缘向右；右滚转时右翼副翼上、左翼副翼下。

视觉舵面不是装饰，它是坐标和符号的调试仪器。动画方向错往往预示物理控制符号也值得重新检查。

相机步骤：从固定跟随到三模式

最初只需 eye = pos - forward*distance + worldUp*height。 然后对 eye 和 target 做插值，消除物理高频抖动。座舱相机改用机体 up， 追逐相机仍用世界 up，避免飞机滚转时画面完全翻转。

HUD 步骤 1：先画固定准星

在 2D Canvas 中每帧先清屏并按 DPR 设置变换。画布 CSS 尺寸和实际像素尺寸不同， 忽略 DPR 会让高分屏文字模糊。

HUD 步骤 2：人工地平线

将 Canvas 原点移到屏幕中心，按负滚转角旋转，再按俯仰角上下平移刻度线。 最后恢复变换并绘制固定飞机符号。这样动的是“世界地平线”，不是飞机符号。

HUD 步骤 3：滚动带

对速度带，遍历当前速度附近的整十刻度，把差值乘像素比例得到 Y 坐标； 用 clip 限制在带框内，再覆盖一个固定当前值窗口。高度带同理。

HUD 步骤 4：小地图缓存

离屏 Canvas 只需几十像素。每个像素采样一次 groundHeight 并按高度着色， 放大绘制到 HUD。飞机每跨 500 m 才重建底图；跑道、航迹和飞机标记每帧叠加。

声音步骤：先单振荡器，再塑造频谱

  1. 用户首次点击后创建 AudioContext，避免自动播放策略拦截。
  2. 用一个振荡器确认声音链路可用。
  3. 叠加低频正弦提供厚度、三角波提供主体、少量锯齿提供机械质感。
  4. 使用低通和 highshelf 压低刺耳高频。
  5. 用 setTargetAtTime 平滑参数，避免每帧跳变产生咔哒声。

动手练习

临时只保留锯齿波，再逐个恢复低频正弦、三角波和滤波器。 用耳朵辨认每个节点对音色的贡献，而不是只复制参数。

程序机模

固定翼和四轴都由盒体组合，不依赖外部模型。静态部分合并为一个 Mesh， 可动舵面和旋翼单独绘制。绕任意铰链点旋转使用：

M = T(pivot) · R(axis,angle) · T(−pivot)

先把铰链移到原点，旋转，再移回去。舵面几何必须先放在真实机体位置， 否则会围绕远处铰链画出错误圆弧。

三种相机

  • 追逐：根据机头水平投影放在后上方，并用插值形成弹簧感。
  • 座舱：相机跟随机体姿态，右摇杆改变观察方向。
  • 环绕：相机沿世界坐标圆周运动，摇杆调整速度和高度。

HUD

HUD 使用独立 2D Canvas，避免为文字和仪表编写 WebGL 字体系统。 姿态梯随飞机滚转和俯仰变换，速度/高度带围绕当前值滚动。 小地图缓存低分辨率地形，只在中心跨过 500 m 时重建，减少每帧噪声采样。

声音合成

WebAudio 使用锯齿、三角和正弦振荡器叠加发动机频谱，经低通和高频搁架滤波， 再按转速改变频率、截止频率和音量。四轴使用更高基频，零油门时真正静音； 失速告警是独立方波振荡器，通过增益门控形成间歇蜂鸣。

let failures = 0;
function check(name, condition, detail) {
  console.log(condition ? '[PASS]' : '[FAIL]', name, detail);
  if (!condition) failures++;
}

项目不需要完整测试框架也能得到稳定价值。关键是测试进程失败时返回非零退出码， 让脚本或 CI 能发现问题。

步骤 3：从用户抱怨反推验收条件

主观反馈可测指标

“爬升率太低”巡航速度下推力余功率对应的理论爬升率

步骤 4：不要只测“方向正确”

方向舵测试若只断言 r > 0，即使响应大到每秒旋转十圈也会通过。 因此项目同时要求它小于合理上限。测试区间应表达可接受行为带，而不是精确浮点值。

步骤 5：调参工作流

  1. 先运行测试记录基准。
  2. 只修改一组相关参数。
  3. 重新运行全部测试，而不仅是当前目标测试。
  4. 在浏览器实际试飞，确认数值通过不等于手感必然优秀。
  5. 若发现新回归模式，先补测试，再修代码。

src/render/scene.js

实现顺序：单着色器画一个 Mesh → 加地形/跑道 → 先画天空 → 加光照雾 → 构造飞机世界矩阵 → 分派机型。验收：每帧 viewport 与 Canvas 一致；天空绘制后恢复深度状态；零件矩阵不会污染下一个 draw。

src/render/aircraft-mesh.js

实现顺序：机身 → 机翼尾翼 → 起落架 → 独立舵面 → 铰链矩阵 → 螺旋桨。每加一个舵面先只输入 ±0.2 检查方向和旋转中心。验收：所有几何位于真实机体局部坐标，右方向舵后缘向右，右翼副翼在右滚时向上。

src/render/quad-mesh.js

实现顺序：中心盒 → 两根机臂 → 前向颜色块 → 电机 → 独立旋翼。验收：前向标记位于 +X；四电机围绕中心对称；相邻旋翼视觉旋向相反；零油门时旋翼停止。

界面与声音

src/ui/hud.js

实现顺序：DPR resize → 固定准星 → 姿态梯 → 三个滚动带 → 遥测文字 → 遮罩/帮助 → 小地图。每个绘制函数都用 save/restore 隔离变换。验收：滚转时仅地平线旋转；不同分辨率文字清晰；地图更新不造成明显帧顿。

src/ui/audio.js

实现顺序：用户手势后创建 context → 单振荡器 → gain → 多振荡器 → 滤波 → 随转速平滑参数 → 告警通道。验收：初次加载不报 autoplay 错误；四轴零油门静音；状态变化没有明显咔哒爆音。

测试

test/physics-smoke.mjs

编写顺序：先建 flat ground 和 check 工具；每个 IIFE 创建全新状态；按真实秒数循环固定步；输出关键数值。验收：故意降低推力、翻转舵面或删除失速混合时，对应测试必须失败。

test/quad-smoke.mjs

编写顺序：创建 airborne helper → 理论悬停 → 高油门爬升 → 单轴滚转。测试阈值留出积分误差，不比较逐位相等。验收：修改质量后若不调整悬停公式，测试能指出漂移。

test/gamepad-smoke.mjs

编写方法：不用真实浏览器，构造假的 pads、axes 和 buttons 输入纯函数。验收：数组含 null 不抛异常；RB 时 trim 非零、look 为零、左摇杆舵量仍保留。

test/terrain-smoke.mjs

编写方法：用最小 1×1 网格取得前三个索引，手工计算两条边的二维叉积。验收：交换任意两个索引后测试应立即失败，证明它真正保护绕序。

CHAPTER 12

推荐学习与改造路线

本章目标

  • 把前面内容转化为可以实际执行的学习计划。
  • 知道每个阶段何时算“完成”，避免永远停留在看懂代码。
  • 选择一个适合自己的后续功能练习。

建议按 12 次小提交完成

  1. 页面与蓝色清屏

    提交标准：本地服务器打开后，WebGL Canvas 能稳定铺满窗口。

  2. 第一个三角形与 Mesh

    提交标准：能用同一 Mesh 类画两个不同位置的盒子。

  3. 数学与相机

    提交标准：相机可绕原点，四元数旋转结果有简单断言。

  4. 地形与跑道

    提交标准：飞机位置改变时地形窗口更新且没有裂缝。

  5. 输入协议

    提交标准：键盘和假手柄产生同形状对象。

  6. 重力、推力和升阻力

    提交标准：空中初始状态能滑翔，不再像石头下落。

  7. 角运动和舵面

    提交标准：三轴输入方向正确，松杆后阻尼合理。

  8. 起落架

    提交标准：从跑道起飞并可轻柔接地。

  9. HUD、相机与声音

    提交标准：无需看控制台即可判断速度、高度、油门和告警。

  10. 四轴模式

    提交标准：同一输入协议可切换，并在理论油门附近悬停。

  11. 行为测试

    提交标准：关键手感都有数值边界，故意破坏会失败。

  12. 文档与清理

    提交标准：新读者能从 README 启动，从本教材复刻。

  1. 先画静态世界：只保留 gl.js、shader.js、mesh.js、scene.js，理解渲染管线。
  2. 让物体移动：阅读 math.js 和 main.js，掌握固定步长和四元数插值。
  3. 加入简化飞行：先实现重力与推力，再逐项加入升力、阻力和力矩。
  4. 加入接地：从单接触点开始，再扩展到三轮起落架和侧向摩擦。
  5. 加入第二机型：仿照 quad.js 保持输入协议不变，只替换动力学解释。
  6. 最后调手感：每次调参都运行测试，并把新的主观要求转成数值断言。
一个很实用的原则

渲染可以近似，物理可以简化，但坐标系、符号约定和状态所有权必须明确。大多数“方向反了”与“模型飞走了”都源于这三件事。
